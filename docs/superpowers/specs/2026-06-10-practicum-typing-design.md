# Practicum — typing-практикум з diff кодового агента

**Дата:** 2026-06-10
**Статус:** реалізовано. v2 (того ж дня, рішення користувача): дірки
прибрано, додано покрокові пояснення блоків; візуальний дизайн «Graphite»
з Claude Design.

## Проблема

Повний перехід на написання коду агентом зупинив розвиток практичної навички
програмування: архітектурне розуміння росте, а здатність самостійно написати чи
проревьювити код — ні. Дослідження Anthropic (02.2026) підтверджує ефект:
AI-асистування знижує засвоєння навичок (~17%).

Існуючі рішення закривають лише частини:

- typing.io, monkeytype code mode — чистий друк, без пояснень і без зв'язку з
  власним проєктом;
- Claude Code Learning/Explanatory output styles — пояснення в потоці роботи та
  `TODO(human)`, але не структурований урок після чекпоінта;
- Codeamigo — друк з AI-підказками, але генеричні уроки, проєкт покинуто;
- Diffity — пояснення по diff без typing-механіки.

Ніша «агент закінчив чекпоінт → з його diff генерується практикум з поясненнями
→ передрук з активним пригадуванням» — вільна.

## Рішення (один абзац)

Slash-команда `/practicum` у Claude Code: агент бере diff чекпоінта, відбирає
3–7 найцінніших для навчання фрагментів, до кожного пише просте пояснення
українською і розбиває код на логічні блоки з «дитячими» нотатками, та генерує
самодостатній HTML-файл із monkeytype-подібним тренажером. Користувач
передруковує код посимвольно; під кодом з'являється нотатка блоку, який він
друкує просто зараз.

## Зафіксовані вимоги

| Питання | Рішення |
|---|---|
| Аудиторія | Особистий інструмент (продукт — можливо потім) |
| Механіка | Передрук + покрокові нотатки блоків (v1 мав дірки/active recall — прибрані у v2) |
| Платформа | Плагін Claude Code + локальний веб-UI |
| Тригер | Вручну командою `/practicum` |
| Обсяг уроку | Відбір агентом, ~2500–4000 символів коду ≈ 10–20 хв |
| Мова пояснень | Українська, рівень початківця (2–4 речення на фрагмент) |

## Архітектура

Репозиторій `~/Programming/practicum` — сорси; встановлення — симлінки в
`~/.claude/`:

```
practicum/
├── commands/practicum.md        → ~/.claude/commands/practicum.md
├── skills/practicum/
│   ├── SKILL.md                 → ~/.claude/skills/practicum/SKILL.md
│   └── template.html            → ~/.claude/skills/practicum/template.html
├── sample-lesson.json           (golden-урок для ручного тесту UI)
└── docs/superpowers/specs/      (ця спека)
```

- **`commands/practicum.md`** — слеш-команда: приймає опціональний аргумент
  (діапазон комітів, напр. `HEAD~3..`; за замовчуванням — незакомічені зміни,
  якщо їх нема — останній коміт), інвокує skill.
- **`skills/practicum/SKILL.md`** — правила: як збирати diff, критерії відбору
  фрагментів, як писати пояснення, де ставити дірки, формат JSON, як вшити JSON
  у копію template.html і відкрити браузер.
- **`skills/practicum/template.html`** — увесь UI (vanilla JS + CSS) з
  плейсхолдером `__LESSON_JSON__`. Жодних залежностей, серверів, білдів.

Результат роботи команди: `.practicum/lessons/YYYY-MM-DD-<тема>.html` у
поточному проєкті (додається в `.gitignore` проєкту за бажанням користувача).

## Потік даних

1. Користувач: `/practicum [діапазон]`.
2. Агент збирає diff (`git diff` / `git show`), читає контекст змінених файлів.
3. Відбір фрагментів (3–7 шт., сумарно 2500–4000 символів коду):
   - пріоритет: нова логіка > патерни/ідіоми > решта;
   - пропускати: імпорти, конфіги, boilerplate, механічні повтори.
4. Для кожного фрагмента:
   - пояснення українською, 2–4 речення, рівень початківця: «що це» і «навіщо
     саме тут»;
   - покрокові нотатки: код розбивається на логічні блоки по 1–5 рядків,
     кожен блок отримує 1–2 речення «як дитині» — що пишемо і навіщо.
5. Згенерувати lesson-JSON, підставити в копію `template.html` замість
   `__LESSON_JSON__`, зберегти в `.practicum/lessons/`, відкрити `xdg-open`.

## Формат уроку

```json
{
  "title": "Кешування статистики користувача",
  "project": "django-rail-ai",
  "date": "2026-06-10",
  "fragments": [
    {
      "file": "app/views.py",
      "explanation": "Кешуємо результат, щоб не рахувати двічі...",
      "code": "def get_stats(user_id: int) -> Stats:\n    ...",
      "steps": [
        {
          "start": 0,
          "end": 38,
          "note": "Оголошуємо функцію: приймає id користувача, поверне статистику."
        }
      ]
    }
  ]
}
```

Кроки — суміжні діапазони офсетів у `code`, вирівняні по рядках, покривають
весь фрагмент. UI лише рендерить; уся «розумна» робота — на боці агента.

## UI (template.html)

- Одна сторінка: прогрес по фрагментах, активний фрагмент — пояснення зверху,
  код нижче.
- Передрук посимвольний: правильний символ — зелений, помилка — короткий
  червоний спалах рамки коду (курсор не рухається), курсор на поточній
  позиції (як monkeytype). Enter відтворює перенос рядка,
  початкові відступи рядка вводяться автоматично (щоб не друкувати пробіли).
- Покрокова нотатка: під кодом панель «зараз пишемо» з поясненням блоку,
  в якому стоїть курсор; змінюється, коли друк переходить у наступний блок.
- Завершення: точність (%), час, помилки друку. Історія сесій — localStorage.

## Обробка помилок і межі

- Нема git-репо чи порожній diff → агент повідомляє і не генерує урок.
- Великий diff → ліміт 2500–4000 символів усе одно діє (відбір жорсткіший).
- Таби нормалізуються у 4 пробіли при генерації; довгі рядки — горизонтальний
  скрол, автоперенос вимкнено.
- Юнікод у межах BMP друкується як є; emoji та символи поза BMP у
  фрагменти не відбираються (обмеження двигуна друку).

## Тестування і критерій успіху

- `sample-lesson.json` — golden-урок у репо; на ньому вручну перевіряється
  template.html: друк, помилки, дірки, підказки, екран метрик.
- Інтеграційно: запустити `/practicum` на реальному чекпоінті будь-якого
  проєкту.
- **Критерій успіху MVP:** пройти один реальний урок від виклику `/practicum`
  до екрану метрик без багів.
- Автотестів на MVP немає — свідоме рішення для особистого однофайлового
  інструмента.

## Поза межами MVP (свідомо)

- Локальний сервер, бібліотека уроків, історія у файлах, spaced repetition
  (природна еволюція «підхід B» — формат JSON сумісний).
- Автогенерація після чекпоінта (hook) — можна додати пізніше.
- Рівні складності («сходинки»: передрук → дірки/active recall → написати
  самому). Дірки v1 прибрані у v2 — можуть повернутись як окремий рівень.
- Публічний продукт, підтримка інших агентів, онбординг.
