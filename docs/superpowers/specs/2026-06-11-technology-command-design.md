# Дизайн: команда /vibe-typing:technology

Дата: 2026-06-11. Статус: затверджено користувачем.

Навчальний курс по одній технології в контексті поточного проекту:
як вона інтегрована (або як її інтегрувати) — from scratch, уроками
typing-тренажера. Усе крутиться навколо навчання.

## Команда і диспетчеризація

Навичка `skills/technology/SKILL.md`, виклик `/vibe-typing:technology [назва]`.

1. Без аргументу → режим «інвентаризація».
2. З аргументом: перевірити, чи технологія реально використовується
   (файли залежностей + імпорти + конфіги + docker-compose) →
   режим «є» або режим «нема».
3. Неоднозначна назва (redis-сервіс vs бібліотека aioredis) — перепитати.

## Режим «інвентаризація» (без аргументу)

- Скан: pyproject/requirements/package.json тощо, docker-compose,
  конфіги, ключові імпорти.
- Вивід у чат: список технологій, по рядку «що це і де воно у нас»,
  з позначкою, чи вже існує курс (`.practicum/course-<tech>.md`).
- Стоп. Жодних файлів не пише — користувач сам викликає
  `/technology <обрана>`.

## Режим «технологія Є» (`/technology redis`)

- Механіка how-to з фокусом на наскрізну технологію.
- Стан: окремий `.practicum/course-<tech>.md` (формат course.md
  з how-to, з етапами) — не конфліктує з основним курсом.
- План еволюційний: «як би ми додавали цю технологію з нуля»:
  конфіг/env → ініціалізація (клієнт/двигун) → перше використання →
  решта патернів (кеш, локи, dedup — кожен своїм уроком/етапом).
- Ті ж правила, що в how-to: інкременти, реальні рядки фінальних
  файлів, знахідки про криве використання, підтвердження плану.
- Уроки: `tech-<tech>-NN-<slug>.html`, орієнтир 3–10 уроків.
- Кожен наступний виклик — наступний `- [ ]` урок.
- Технологія згадана тривіально (1–2 рядки) → чесно попередити,
  що курс буде закоротким, і спитати чи продовжувати.

## Режим «технології НЕМА» (`/technology celery`) — інтеграційний курс

Принцип: вчитись можна лише на коді, що точно працює. Тому інтеграція
реальна, в окремій гілці, з перевіркою.

1. План інтеграції: мінімальне робоче ядро технології в ЦЬОМУ проекті
   (залежність → конфіг → ініціалізація → 1–2 реальні точки
   використання, не «обвішати все»), список файлів, спосіб перевірки.
2. ЯВНЕ ЗАСТЕРЕЖЕННЯ перед стартом — користувач має чітко розуміти,
   що зараз станеться: це не просто генерація уроку, а повноцінна
   реалізація; буде створено гілку і написано реальний код; це може
   бути тривалим і витратити багато токенів. ДОЧЕКАТИСЬ явної згоди.
3. Брудне робоче дерево → стоп («закоміть або стешни»). Чисте →
   гілка `vibe-typing/<tech>` від HEAD.
4. Інтеграція в гілці: код → самостійне рев'ю → тести проекту →
   smoke-перевірка. У звіті чесно: що перевірено, що ні.
5. Коміт у гілці, повернення на вихідну гілку — робоче дерево
   користувача не чіпаємо.
6. Уроки з diff `<вихідна>..vibe-typing/<tech>` (механіка practicum +
   еволюційні етапи), стан у `course-<tech>.md` з позначкою гілки.
7. Після останнього уроку — вибір: змерджити гілку (код готовий) або
   видалити і внести зміни руками за уроками (максимум навчання).

## Спільне

- `lesson-format.md` реюзається без змін (skips, офсети, валідатор,
  збирання HTML).
- README: документувати команду і три режими (для режиму «нема» —
  згадати, що це повноцінна реалізація за згодою).
- Перевірка фічі: реальний прогін `/technology redis` на
  ai-post-generated-bot (Redis там є) + прогін інвентаризації.
